# WHMCS 9 Native Alternatives to Lagom Systems

This document maps Lagom systems to WHMCS 9 / Nexus-native features. The goal is to decide what to build natively instead of copying Lagom's addon-heavy implementation.

## Summary table

| Lagom system | WHMCS 9-native alternative | Recommendation |
| --- | --- | --- |
| RSThemes addon UI | Theme hooks + small settings service | Use a small custom addon only if admin editing is required |
| Runtime page manager | Template fallback + filesystem variants | Use filesystem variants for MyTheme |
| Menu builder | `ClientAreaPrimaryNavbar` / `ClientAreaSecondaryNavbar` hooks | Prefer code-driven menus |
| Sidebar builder | `ClientAreaPrimarySidebar` / `ClientAreaSecondarySidebar` hooks | Prefer code-driven sidebars |
| Dashboard widgets | Native `$panels` + `ClientAreaHomepagePanels` | Prefer native panels |
| Style presets | CSS variables / `custom.css` / generated CSS | Use simple preset token files |
| Layout selector | Smarty include paths + body classes | Use simple layout setting |
| Dynamic CSS injection | `ClientAreaHeadOutput` | Use when settings must generate CSS |
| Dynamic JS injection | `ClientAreaFooterOutput` | Use scoped page/module scripts |
| Page SEO manager | `ClientAreaPage*` hooks + header SEO partial | Use lightweight hooks |
| CMS/custom pages | Filesystem templates or WHMCS-friendly page module | Avoid a broad page DB unless needed |
| Order form settings | Normal WHMCS order form templates | Keep close to native cart unless UX requires custom cards |
| Module integrations | Head/Footer output hooks | Use scoped integration hooks |

## 1. Dashboard panels

### Lagom approach

Lagom can replace native homepage panels with an `RSWidgets` dashboard generated by the addon.

### WHMCS 9-native approach

Use native dashboard panels:

- `{$panels}` in `clientareahome.tpl`
- `ClientAreaHomepagePanels` hook to add/remove/reorder panels
- `MenuItem` methods such as `getName()`, `getChildren()`, `getLabel()`, `getUri()`, `getBadge()`, `getBodyHtml()`, and `getFooterHtml()`

### Recommendation

For MyTheme, prefer native `$panels` and render them in Apple-style cards. This preserves compatibility with WHMCS 9 and third-party modules.

Use custom/localAPI dashboard data only as fallback.

## 2. Menu and sidebar customization

### Lagom approach

Database-backed admin UI for menus and sidebars.

### WHMCS 9-native approach

Use hooks:

```php
add_hook('ClientAreaPrimaryNavbar', 1, function (WHMCS\View\Menu\Item $menu) {
    $menu->addChild('custom')
        ->setLabel('Custom')
        ->setUri('/custom')
        ->setOrder(100);
});
```

### Recommendation

Use hooks for static or developer-maintained navigation.

Only build an admin menu editor if the customer must edit navigation without code.

## 3. Page variants

### Lagom approach

Page metadata and active variant are resolved from RSThemes page settings and rendered via `$RSThemes['pages'][$templatefile]['fullPath']`.

### WHMCS 9-native approach

Use filesystem-based variants:

```text
templates/mytheme/core/pages/clientareahome/default/default.tpl
templates/mytheme/core/pages/clientareahome/compact/compact.tpl
templates/mytheme/core/pages/login/sidebar/sidebar.tpl
```

Then use a small dispatcher:

```text
if configured variant exists:
    include variant
else:
    include default
```

### Recommendation

This is already close to MyTheme's architecture. Keep it simple and version-controlled.

## 4. Styling presets

### Lagom approach

Style presets live under `core/styles/*` and are selected by addon settings. Runtime CSS can be injected from style color objects.

### WHMCS 9-native approach

Use CSS custom properties:

```css
:root {
    --color-accent: #0071e3;
    --color-surface: #ffffff;
}

[data-theme="dark"] {
    --color-surface: #1c1c1e;
}
```

Preset files can be small token layers:

```text
assets/css/presets/default.css
assets/css/presets/depth.css
assets/css/presets/futuristic.css
assets/css/presets/modern.css
```

### Recommendation

Implement presets as token files and one active body class or stylesheet. Avoid a large runtime style compiler unless the admin UI requires live editing.

## 5. Layout selection

### Lagom approach

`header.tpl` includes `$RSThemes['layouts']['mediumPath']`, while `footer.tpl` includes `$RSThemes['footer-layouts']['mediumPath']`.

### WHMCS 9-native approach

Use normal Smarty includes:

```text
core/layouts/main-menu/default/default.tpl
core/layouts/main-menu/left-nav/left-nav.tpl
core/layouts/footer/default/default.tpl
core/layouts/footer/extended/extended.tpl
```

Set the active layout with a small config value or hook variable.

### Recommendation

Adopt this concept, but avoid database-heavy layout resolution.

## 6. Head/footer output and module integration

### Lagom approach

Module integrations return CSS/JS through `ClientAreaHeadOutput` and `ClientAreaFooterOutput` after RSThemes service processing.

### WHMCS 9-native approach

Use the same hooks directly:

- `ClientAreaHeadOutput` for `<link>` / `<style>`
- `ClientAreaFooterOutput` for `<script>`

### Recommendation

Keep integration logic small, scoped, and page/module-aware.

## 7. SEO and CMS metadata

### Lagom approach

Page SEO is stored in addon-backed page models and returned to templates through hooks.

### WHMCS 9-native approach

Use a header SEO partial plus hook variables:

```php
add_hook('ClientAreaPageAnnouncements', 1, function ($vars) {
    return [
        'pageSeo' => [
            'title' => 'Announcements',
            'description' => 'Latest hosting news',
        ],
    ];
});
```

### Recommendation

Use lightweight hook-provided metadata unless a full CMS is required.

## 8. Order form features

### Lagom approach

Order form templates are highly configurable through `$RSThemes['pages'][$templatefile]['config']` and addon-calculated pricing variables.

### WHMCS 9-native approach

Use a normal order form theme with WHMCS-provided variables and limited custom settings.

### Recommendation

Do not copy Lagom's pricing and product-card complexity unless required. WHMCS 9/Nexus already provides stable cart variables and template separation.

## 9. Addon settings storage

### Lagom approach

Many custom tables and addon models.

### WHMCS 9-native approach

Use:

- `theme.yaml` for theme metadata and dependencies.
- `tblconfiguration` for small global settings.
- A single custom settings table only if necessary.
- Files for version-controlled config.

### Recommendation

For MyTheme, keep the existing small settings model. Avoid many tables for pages/menus/widgets unless the admin UI truly needs them.

## 10. What WHMCS 9 gives us already

WHMCS 9 / Nexus already provides:

- Template fallback and child-theme-style overrides.
- `theme.yaml` dependency declaration.
- `assetPath` and `assetExists` helpers.
- Standard `$headoutput`, `$headeroutput`, `$footeroutput` hooks.
- `hookOutput` and `addons_html` injection points.
- `MenuItem` API for nav/sidebar/panels.
- Native client homepage `$panels`.
- `ClientAreaPage*` hooks for page variables.
- Standard cart/order form template separation.
- Standard OAuth, payment, PDF, store, and error template structures.

Use these first. Add custom systems only where WHMCS does not already provide a stable native extension point.
