# Architecture

> **Target stack**: WHMCS 9.0+, PHP 8.1+, Smarty 4 (`\Smarty\Smarty` namespace) or Smarty 3 (`\Smarty`), ionCube 13+. The codebase uses `readonly` properties, backed enums, named arguments, `match` expressions, and `never` return types.

## Two-piece product

```
modules/addons/MyTheme/   ← engine (encoded for production)
templates/mytheme/        ← visual layer (mostly plain)
```

- The **addon** holds business logic: license check, hooks dispatch, admin UI, DB models.
- The **template** holds presentation: Smarty tpls, SCSS sources, assets, per-variant manifests.

Either piece is useless without the other. WHMCS picks the template by name from `tblconfiguration.Template`. The addon validates the license, populates the `$myTheme` Smarty super-array, and force-resets the template to `six` if licensing fails.

## Discovery model

**No filesystem scans at runtime.** A theme's capabilities are declared once in `templates/<slug>/theme.json`:

```json
{
  "provides": {
    "styles": ["default", "dark"],
    "layouts": { "main-menu": ["top", "sidebar"], "footer": ["default"] },
    "pages": ["clientareahome", "login", ...],
    "extensions": []
  }
}
```

The engine reads it once at construction and caches in-process. Adding a new style is: (1) drop the directory, (2) add its name to the manifest, (3) bump version. The Configuration cache is invalidated automatically on version change.

## Dispatch model — Smarty 3 inheritance pattern

Lagom's pattern: `{if file_exists(...)}{include}{else}<300 lines of inline default>{/if}` inside every WHMCS-named root tpl.

This pattern: 5-line dispatcher, default lives at `core/pages/<page>/default/default.tpl`.

```smarty
{* templates/mytheme/clientareahome.tpl *}
{if isset($myTheme.pages.clientareahome.fullPath) && file_exists("templates/`$myTheme.pages.clientareahome.fullPath`")}
    {include file="`$myTheme.pages.clientareahome.fullPath`"}
{else}
    {include file="`$template`/core/pages/clientareahome/default/default.tpl"}
{/if}
```

The "default" variant is a real file with a real implementation. The dispatcher just picks between configured variant and default. No 300-line `else` branches in the WHMCS-named entry points.

## Hook architecture

ONE `ClientAreaPage` registration → `HookDispatcher::dispatch()` → routes to a method on `Service\Hooks`.

```php
// hooks.php — ONE registration per WHMCS hook point
add_hook('ClientAreaPage', -1, fn($vars) =>
    Hooks::instance()->dispatch('ClientAreaPage', $vars));

// src/Service/Hooks.php — one method per hook point
private function clientAreaPage(array $vars, Template $template): array { ... }
private function clientAreaHeadOutput(array $vars, Template $template): ?string { ... }
private function clientAreaFooterOutput(array $vars, Template $template): ?string { ... }
```

Compare to Lagom: 6 separate `ClientAreaPage` hook registrations with mixed priorities, anonymous closures next to named functions, a 200-line god-function (`RSThemeCAVars`).

## Settings model — two stores, not four

| Store | Used for |
|---|---|
| `tblconfiguration` | License state (key, blob, warning, hour, domain), WHMCS-native template selection (`Template`, `OrderFormTemplate`), `ActiveAddonModules` flag |
| `mytheme_settings` | Everything else — typed key/value (string/int/bool/json) for theme-specific addon settings |

Lagom uses four (`tbladdonmodules` + `tblconfiguration` + `rsthemes_settings` + JSON-in-text in `rstheme_themes`). Don't.

## License client — improved over typical commercial themes

| Aspect | This client |
|---|---|
| Local cache MAC | HMAC-SHA256, `hash_equals`, `json_encode` payload |
| Server response signing | RSA-SHA256, embedded public key |
| Replay protection | Per-request nonce + `signed_at` recency check (≤24h) |
| TLS | `CURLOPT_SSL_VERIFY*` enabled |
| State machine | `LicenseState` class (not stringly-typed) |
| Banner UI | Smarty .tpl files (not heredoc'd HTML in PHP) |
| Code organization | License client + cache + crypto + state separated |

Threat model and setup details: see [LICENSING-SETUP.md](../LICENSING-SETUP.md).

## Anti-tamper — chained sha256 stubs with build-time codegen

Every protected file calls `IntegrityHashes::verifyOrDie(__FILE__)` in its constructor. The hash table is generated by `scripts/inject-integrity-hashes.mjs` at build time — never hand-written.

```php
// src/Helpers/IntegrityHashes.php (GENERATED)
private const HASHES = [
    'modules/addons/MyTheme/src/Template/License.php'  => 'a3f2...',
    'modules/addons/MyTheme/src/Template/Template.php' => 'b9e1...',
    // ...
];
```

Patching `License.php` causes `Template.php`'s integrity check to fail (and vice versa) — no single point of disarm. Every new protected file added to `scripts/integrity-targets.json` adds another guard.

## Smarty 4 compatibility (WHMCS 9)

WHMCS 9 ships Smarty 4 in the `\Smarty\Smarty` namespace. Earlier WHMCS used Smarty 3 as `\Smarty`. `AbstractController::resolveSmartyClass()` picks whichever is loaded:

```php
public static function resolveSmartyClass(): string
{
    return match (true) {
        class_exists('\\Smarty\\Smarty') => '\\Smarty\\Smarty',
        class_exists('\\Smarty')         => '\\Smarty',
        default => throw new \RuntimeException('Smarty engine not available'),
    };
}
```

This keeps the addon working on legacy WHMCS 8 installs without forking. When you drop WHMCS 8 support, hardcode `\Smarty\Smarty` and remove the resolver.

## Why not Smarty `{extends}/{block}` for the page dispatch?

Considered. The simpler `{include}` dispatch is preferred because:

1. Variant tpls become shareable across pages without inheritance trees
2. The pattern works identically on Smarty 3 and 4 (no version-specific syntax)
3. No unexpected block-resolution surprises when buyers add their own variants

`{extends}` is the right tool when variants share large structure with small overrides. Most WHMCS pages don't — each is mostly different markup.

## What's deliberately NOT in the v1 starter

- Per-vendor `store/<name>/` integrations (do these as separate addons)
- Multi-display CMS routing
- Drag-and-drop menu builder
- Sidebar builder
- Page builder
- Dynamic widgets system
- Color picker that recompiles SCSS at runtime

These are products on top of the engine. Build them once the engine is proven.

## File-naming conventions

| Kind | Convention | Examples |
|---|---|---|
| PHP class | PascalCase, namespace per directory | `License.php`, `LicenseState.php` |
| PHP config | lowercase or kebab-case, returns array | `theme.json`, `style.php`, `layout.php`, `page.php`, `pageoption.php` |
| Smarty page | kebab-case, matches WHMCS template name | `clientareahome.tpl`, `password-reset-container.tpl` |
| Smarty partial | kebab-case | `main-menu.tpl`, `top-nav.tpl` |
| Variant directory | lowercase, single word preferred | `default/`, `dark/`, `top/`, `sidebar/` |
| SCSS partial | underscore-prefixed | `_variables.scss`, `_components.scss` |
| Build script | kebab-case | `build-css.mjs`, `inject-integrity-hashes.mjs` |
